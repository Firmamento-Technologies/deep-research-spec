## 32.1 Config Wizard

Four sequential steps. No step is skippable; each validates before advancing.

```python
class WizardStep(TypedDict):
    id: Literal["topic","style","sources","budget"]
    title: str
    fields: list[dict]  # see per-step schemas below

class TopicStep(TypedDict):
    topic: str                          # 10–500 chars
    target_words: int                   # 1000–50000
    output_language: str                # BCP-47

class StyleStep(TypedDict):
    style_profile: Literal[
        "scientific_report","business_report","technical_documentation",
        "journalistic","narrative_essay","ai_instructions","blog"
    ]
    quality_preset: Literal["economy","balanced","premium"]
    output_formats: list[Literal["docx","pdf","markdown","latex","json"]]
    # Live style preview: system renders 80-word sample paragraph from topic
    # using selected profile; updates on profile change, debounce 600ms

class SourcesStep(TypedDict):
    enable_academic: bool
    enable_institutional: bool
    enable_web: bool
    enable_social: bool
    uploaded_source_ids: list[str]      # POST /v1/sources first; see §24
    source_whitelist: list[str]         # domain strings
    source_blacklist: list[str]

class BudgetStep(TypedDict):
    max_budget_dollars: float           # ≥1.00
    # UI renders three estimates (read-only):
    # optimistic_usd: 1.0 iter/section
    # median_usd:     2.5 iter/section
    # pessimistic_usd: max_iter/section
    # Color: green if max_budget >= median, amber if median<=max<pessimistic,
    #        red if max_budget < median
```

**Inline validations** (client-side, before API call):

| Field | Rule | Error |
|---|---|---|
| `topic` | len ∈ [10,500] | "10–500 characters required" |
| `target_words` | ∈ [1000,50000] | "Must be 1000–50000" |
| `max_budget_dollars` | ≥ median_estimate | "Budget below median estimate — runs may stop early" (amber warning, not block) |
| `output_formats` | len ≥ 1 | "Select at least one format" |

**Style Preview Widget**: `GET /v1/style-preview` → 200 with `{sample_text: str}`. Renders inline, tagged "Example output". Regenerates on profile or quality_preset change.

## 32.2 Outline Editor

Rendered after Planner produces outline and before production starts (§4.1 await_outline node).

```python
class OutlineSection(TypedDict):
    index: int
    title: str          # editable inline, max 120 chars
    scope: str          # editable inline, max 400 chars
    estimated_words: int
    dependencies: list[int]

class OutlineEditorState(TypedDict):
    sections: list[OutlineSection]
    revision_request: Optional["OutlineRevisionRequest"]
    locked: bool        # True once approved

class OutlineRevisionRequest(TypedDict):
    trigger: OutlineRevisionTrigger
    section_index: int
    message: str        # e.g. "Insufficient sources found for §3 — suggest narrowing scope"
    suggested_scope: Optional[str]
```

**Drag-and-drop**: reorder sections; updates `index` fields; `dependencies` referencing moved section update automatically. Circular dependency → blocked with toast "Circular dependency detected".

**Inline editing**: click title or scope field → contenteditable. `Escape` cancels, `Enter`/blur saves. Max lengths enforced live.

**Delete**: removes section; dependent sections get warning badge "Dependency removed".

**Insufficient-sources revision**: when `OutlineRevisionRequest.trigger == "insufficient_sources"`, a yellow banner appears on that section card with `message` and optional `suggested_scope`. User may: accept suggestion (updates scope), dismiss, or delete section.

**Approve**: `POST /v1/runs/{run_id}/approve` with `{action:"outline_approval", outline: OutlineSection[]}`. Locks editor. See §24.3.

## 32.3 Progress Dashboard

Displayed during active run. Data source: SSE stream `GET /v1/runs/{run_id}/stream` (§24.2) + polling `GET /v1/runs/{run_id}` every 30s as fallback.

```python
class DashboardState(TypedDict):
    run_id: str
    phase: str
    sections: list[SectionCard]
    current_section_idx: int
    cost_accumulated_usd: float
    budget_max_usd: float
    iteration_count_current: int
    max_iterations: int
    eta_minutes: Optional[int]          # None if not computable
    recent_events: list[EventLogEntry]  # last 20

class SectionCard(TypedDict):
    index: int
    title: str
    status: SectionStatus
    css_final: Optional[float]          # None if not yet approved
    css_history: list[float]            # all iterations
    iterations_used: int
    cost_usd: float

class EventLogEntry(TypedDict):
    timestamp: datetime
    event_type: str                     # SSE event name
    section_index: Optional[int]
    message: str                        # human-readable, ≤120 chars
```

**Section Status Strip**: one card per section. Colors: `pending`=grey, `in_progress`=blue pulse, `approved`=green, `failed`=red, `regenerating`=amber pulse.

**CSS Trend Chart**: line chart per approved section; x=section index, y=`css_final` (0.0–1.0). Horizontal reference line at configured `css_approval_threshold`. Updated on each `SECTION_APPROVED` event.

**Cost Gauge**:
- Arc gauge 0 → `budget_max_usd`.
- Color zones: 0–70% green, 70–90% amber, 90–100% red.
- Alarms:
  - 70%: info banner "70% budget used"
  - 90%: warning banner "90% budget — fallback models active" (see §19.4)
  - 100%: error banner "Budget exhausted — run stopped"

**Iteration Counter**: `{iteration_count_current} / {max_iterations}` for current section.

**ETA Widget**: `eta_minutes` from API. Formula: `(approved_sections_avg_duration_s × remaining_sections) / 60`. Displays "~N min remaining" or "Estimating…" if `None`.

**Recent Event Log**: last 20 `EventLogEntry` items, newest-first. Non-scrollable (auto-trims to 20). Timestamps in local timezone.

## 32.4 Escalation Interface

Triggered when `escalation_pending == true` in polling response. Run is paused. UI blocks dashboard interaction and shows modal.

```python
class EscalationPayload(TypedDict):
    escalation_type: EscalationType
    section_index: int
    section_title: str
    detail: str                         # why escalation triggered
    css_history: list[float]            # all iterations for this section
    draft_diff: Optional[str]           # unified diff last two drafts (oscillation only)
    coherence_conflict: Optional["CoherenceConflict"]
    available_actions: list[RegenerateAction]

class CoherenceConflict(TypedDict):
    section_a_index: int
    section_a_excerpt: str
    section_b_index: int
    section_b_excerpt: str
    conflict_description: str
```

**Visual Diff** (oscillation & coherence conflict): rendered as side-by-side or unified diff with additions in green, deletions in red. Uses standard `diff` format from `draft_diff` field.

**Actions per escalation type**:

| `escalation_type` | Available actions |
|---|---|
| `oscillation` | textarea for writer instructions → `add_instructions`; `approve_override` (logs motivation); `change_scope`; `skip` |
| `coherence_conflict` | diff display + `add_instructions` targeting section_b; `approve_override`; `skip` |
| `budget_alarm` | read-only info; `approve_override` to continue with degraded settings; `abort_run` |
| `rogue_judge` | shows judge ID + disagreement rate; `approve_override` to substitute judge; continue |
| `full_rewrite` | Reflector flagged FULL scope (§12.2); `add_instructions` for full rewrite; `skip` |

Response: `POST /v1/runs/{run_id}/approve` with `{action: "escalation_response", escalation_response: {action: RegenerateAction, instructions: str, motivation: str}}`.

## 32.5 Section Regeneration and Versioning

Accessible from any `SectionCard` with `status == "approved"` via "Regenerate" button (visible on hover).

```python
class RegenerationRequest(TypedDict):
    section_index: int
    instructions: str       # required, 10–1000 chars
    preserve_sources: bool  # default True

class RegenerationCostEstimate(TypedDict):
    section_index: int
    estimated_usd: float    # 1 iter × token_estimate × model_prices
    current_budget_remaining_usd: float
    feasible: bool          # estimated_usd ≤ current_budget_remaining_usd
```

**Flow**:
1. User clicks "Regenerate" on approved section card.
2. UI calls `GET /v1/runs/{run_id}/sections/{index}/regen-estimate` → `RegenerationCostEstimate`.
3. Modal shows estimate: "Estimated cost: $X.XX — $Y.YY remaining in budget". If `feasible == false`: amber warning, user may proceed anyway (explicit confirm).
4. User enters `instructions` (required), toggles `preserve_sources`.
5. Confirm → `POST /v1/runs/{run_id}/sections/{index}/regenerate`.
6. Section status → `regenerating`. Production of subsequent sections continues uninterrupted (queue-based; see §32.6).
7. On completion: version N+1 added to section store (§21.1). Version switcher appears on card: "v1 | v2 ▾".

**Version switcher**: dropdown shows all versions with `css_final` and `approved_at`. Selecting a version shows read-only diff vs current. "Restore v{N}" button promotes selected version to active.

## 32.6 Style Override Without Interrupting Production

Allows correcting style of an already-approved section while subsequent sections are still being produced.

```python
class StyleOverrideRequest(TypedDict):
    section_index: int
    style_instructions: str     # 10–500 chars; passed to Style Fixer (§5.10)

class StyleOverrideStatus(TypedDict):
    section_index: int
    status: Literal["queued","processing","awaiting_review","completed","failed"]
    original_content: str
    revised_content: Optional[str]  # available when status == "awaiting_review"
    diff: Optional[str]             # unified diff
```

**Behavior**:
- Override is enqueued; does not pause or affect the active production loop.
- Style Fixer (§5.10) processes asynchronously (max 2 iterations).
- When `status == "awaiting_review"`: UI shows diff panel (side-by-side). User approves or rejects.
- Approve → new version written to Store. Reject → original retained.
- Original version remains active in Store until user approves the override.
- If production completes before override: override still processes from queue.

**Access**: gear icon on approved section card → "Fix style" option. Only one pending override per section at a time.

## 32.7 Run Companion Interface

Always-visible sidebar during active run. Width: 320px, collapsible to icon bar.

```python
class CompanionMessage(TypedDict):
    id: str
    role: Literal["user","companion"]
    content: str
    timestamp: datetime
    section_ref: Optional[int]      # section index this message relates to
    iteration_ref: Optional[int]
    modification_applied: Optional[CompanionModification]

class CompanionModification(TypedDict):
    mod_type: CompanionModType
    parameters: dict                # mod_type-specific params
    requires_confirmation: bool     # see §6.4
    confirmed: Optional[bool]       # None = pending

class ProactiveBadge(TypedDict):
    trigger: str                    # e.g. "css_below_0.70_after_approval"
    message: str                    # ≤ 120 chars
    section_index: Optional[int]
    priority: Literal["info","warning","critical"]
```

**Chat Sidebar Layout**:
- Header: "Run Companion" + model indicator (`gemini-2.5-pro`)
- Badge counter: unread proactive notifications (red dot, number)
- Message thread: `CompanionMessage[]` newest-at-bottom, auto-scroll
- Input field: single-line, `Enter` sends, `Shift+Enter` newline; max 500 chars
- "Clear" button: hides read messages (not deleted; preserved in Run Report)

**Proactive Badge Triggers** (see §6.5):

| Trigger condition | `priority` | Message template |
|---|---|---|
| section `css_final` < 0.70 after approval | `warning` | "§{N} approved with low CSS ({v}) — consider regenerating" |
| `cost_accumulated / budget_max` > 0.60 at section ≤ `total/2` | `warning` | "60% budget used at midpoint — check cost gauge" |
| same source_id appears in > 3 consecutive sections | `info` | "Source '{title}' dominates last 3 sections — diversity may be low" |
| Judge F finds external contradiction (§8.2.1) | `warning` | "Judge F found contradicting evidence for §{N} claim" |
| `css_history` std_dev < 0.03 over last 3 iter (oscillation early warning) | `critical` | "§{N} showing convergence stall — escalation likely in next iteration" |
| style drift detected (§16.2) | `info` | "Style drift detected between §{A} and §{B}" |

Badge icon pulses for `critical`, static dot for `warning`/`info`. Clicking badge opens sidebar and scrolls to message.

**Direct modifications** (no confirmation required, see §6.4):
- `add_forbidden_pattern`: companion writes pattern → immediate L1 rule addition for future sections
- `css_threshold_delta`: ±0.10 max per message; clamped to [0.40, 0.95]
- `add_source`, `add_run_note`, `source_blacklist`

**Confirmation-required modifications**: companion proposes → UI shows confirm banner in thread with "Apply" / "Cancel". Run continues during pending confirmation.

**Conversation log**: full `CompanionMessage[]` stored in `companion_messages` field of `DRSState` (§schema); exported in Run Report JSON under `companion_log`. Each applied modification includes `section_ref` + `iteration_ref` for audit traceability.

**API**: `POST /v1/runs/{run_id}/companion` with `{message: str}` → `{reply: CompanionMessage, modification: Optional[CompanionModification]}`. Response latency target: < 3s (§6.6). See §24.
